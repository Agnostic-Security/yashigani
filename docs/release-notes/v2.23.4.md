## v2.23.4 — Reliability Fixes

### BUG-REBOOT-NO-AUTO-START — Containers now auto-start on host reboot

**Risk reference:** ACS-RISK-046 (v2.23.4 TAG-BLOCKER)

Prior to v2.23.4, Yashigani containers did not survive a host reboot. The
`restart: unless-stopped` policy in `docker-compose.yml` was present on all
services, but the underlying Podman or Docker daemon was never configured to
start on boot — so the policy was never exercised after a reboot. A fresh
install produced a completely stopped stack after every host restart, requiring
manual operator intervention to bring services back up.

v2.23.4 adds OS-level auto-start artifacts for all supported runtime classes:

#### Linux rootful Podman (canonical — VM production target)

`install.sh` writes `/etc/systemd/system/yashigani.service` — a oneshot
systemd unit that runs `podman compose up -d` after `network-online.target`
and `podman.socket.service` are ready. The unit is enabled on install
(`systemctl enable yashigani.service`) and appears in the completion summary
so operators can verify the enabled state in the same terminal session.

Dependency ordering: `After=podman.socket.service` + `Requires=podman.socket.service`
ensures the Podman API socket is present before compose attempts to use it.
`Type=oneshot` + `RemainAfterExit=yes` matches the semantics of `compose up -d`:
the start command exits quickly and the service remains active while containers
are running. `ExecStop` tears the stack down cleanly on systemd shutdown.

#### Linux rootless Podman

`install.sh`:
1. Runs `loginctl enable-linger <user>` — this is a hard pre-condition.
   Without linger, the user's systemd instance is killed on logout and
   containers die. With linger, the user's systemd instance starts at boot
   before any login and persists across logouts.
2. Writes `~/.config/systemd/user/yashigani.service` and enables it via
   `systemctl --user enable`.

If `loginctl enable-linger` fails (e.g. on a host where the operator lacks
the capability), install.sh emits a warning and continues — the unit is still
installed and will work once linger is manually enabled. The linger state is
surfaced in the install output so the operator can act immediately.

#### Docker Linux

Docker's own daemon (`docker.service`) typically auto-enables on package
install. `install.sh` now verifies this is the case via `systemctl is-enabled
docker`. If not enabled (unusual but observed on minimal installs), install.sh
enables it automatically and emits a warning. No unit file is written for this
path — `docker.service` + `restart: unless-stopped` provides the same result.

#### macOS Podman (dev-workstation, login-only)

`install.sh` installs `~/Library/LaunchAgents/io.yashigani.autostart.plist`.
The agent fires at user login and runs `podman machine start` (idempotent if
already running) followed by `podman-compose up -d`.

**v2.23.4 limitation:** The LaunchAgent fires at **user login**, not at system
boot. On a developer workstation (the macOS Podman target), this means
Yashigani auto-starts when the admin logs in after a reboot — not before.
An unattended macOS server would require a LaunchDaemon (system-context,
root-owned), which is out of scope for v2.23.4. The boot-time LaunchDaemon
path is deferred to v2.23.5+.

Docker Desktop on macOS manages its own login-item via its system-tray UI —
install.sh does not install a competing LaunchAgent for that path.

#### Kubernetes / Helm

No change. Kubernetes `Deployment`, `StatefulSet`, and `DaemonSet` controllers
restart pods natively on node failure or reboot. The Helm chart requires no
modification.

### Uninstall removes all auto-start artifacts

`uninstall.sh` now removes auto-start artifacts **before** stopping the compose
stack, eliminating the race where a reboot mid-uninstall could re-start the
stack:

- **Linux rootful Podman:** `systemctl disable yashigani.service` + unit file deleted + `daemon-reload`
- **Linux rootless Podman:** `systemctl --user disable yashigani.service` + unit file deleted + `daemon-reload`
- **loginctl disable-linger:** runs only when `--remove-volumes` is passed, as
  part of the full-clean exit path that removes data and linger together. A
  plain `uninstall.sh` preserves linger so a re-install picks up the user
  systemd instance cleanly without requiring manual re-enablement.
- **macOS:** `launchctl unload` + plist deleted
- **Docker Linux:** nothing to remove (no unit was written)
