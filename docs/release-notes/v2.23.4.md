## v2.23.4 — Reliability Fixes

### BUG-REBOOT-NO-AUTO-START — Containers now auto-start on host reboot

**Risk reference:** ACS-RISK-046 (v2.23.4 TAG-BLOCKER)

Prior to v2.23.4, Yashigani containers did not survive a host reboot. The
`restart: unless-stopped` policy in `docker-compose.yml` was present on all
services, but the underlying Podman or Docker daemon was never configured to
start on boot — so the policy was never exercised after a reboot. A fresh
install produced a completely stopped stack after every host restart, requiring
manual operator intervention to bring services back up.

v2.23.4 adds OS-level auto-start artifacts for all supported runtime classes:

#### Linux rootful Podman (canonical — VM production target)

`install.sh` writes `/etc/systemd/system/yashigani.service` — a oneshot
systemd unit that runs `podman compose up -d` after `network-online.target`
and `podman.socket.service` are ready. The unit is enabled on install
(`systemctl enable yashigani.service`) and appears in the completion summary
so operators can verify the enabled state in the same terminal session.

Dependency ordering: `After=podman.socket.service` + `Requires=podman.socket.service`
ensures the Podman API socket is present before compose attempts to use it.
`Type=oneshot` + `RemainAfterExit=yes` matches the semantics of `compose up -d`:
the start command exits quickly and the service remains active while containers
are running. `ExecStop` tears the stack down cleanly on systemd shutdown.

#### Linux rootless Podman

`install.sh`:
1. Checks at pre-flight whether linger is already enabled for the install user
   (`loginctl show-user $USER --property=Linger --value`). If not, it emits a
   copy-pasteable pre-flight warning and pauses 3 seconds so the operator can
   Ctrl-C, run `sudo loginctl enable-linger $USER`, then re-run the installer.
   Linger setup is a documented pre-flight step for rootless Podman installs
   (sysadmin pattern per audience-sysadmins doctrine).
2. Writes `~/.config/systemd/user/yashigani.service` and enables it via
   `systemctl --user enable`.

The installer body no longer attempts `loginctl enable-linger` under `sudo -n`.
That pattern is banned by `feedback_audience_sysadmins` (zero sudo in install
body; sudo silently fails without NOPASSWD) and by the `lint_sudo_pattern.sh`
CI gate (Class B). Linger enablement is operator-side and one-time.

#### Docker Linux

Docker's own daemon (`docker.service`) typically auto-enables on package
install. `install.sh` now verifies this is the case via `systemctl is-enabled
docker`. If not enabled (unusual but observed on minimal installs), install.sh
enables it automatically and emits a warning. No unit file is written for this
path — `docker.service` + `restart: unless-stopped` provides the same result.

#### macOS Podman (dev-workstation, login-only)

`install.sh` installs `~/Library/LaunchAgents/io.yashigani.autostart.plist`.
The agent fires at user login and runs `podman machine start` (idempotent if
already running) followed by `podman-compose up -d`.

**v2.23.4 limitation:** The LaunchAgent fires at **user login**, not at system
boot. On a developer workstation (the macOS Podman target), this means
Yashigani auto-starts when the admin logs in after a reboot — not before.
An unattended macOS server would require a LaunchDaemon (system-context,
root-owned), which is out of scope for v2.23.4. The boot-time LaunchDaemon
path is deferred to v2.23.5+.

Docker Desktop on macOS manages its own login-item via its system-tray UI —
install.sh does not install a competing LaunchAgent for that path.

#### Kubernetes / Helm

No change. Kubernetes `Deployment`, `StatefulSet`, and `DaemonSet` controllers
restart pods natively on node failure or reboot. The Helm chart requires no
modification.

### Uninstall removes all auto-start artifacts

`uninstall.sh` now removes auto-start artifacts **before** stopping the compose
stack, eliminating the race where a reboot mid-uninstall could re-start the
stack:

- **Linux rootful Podman:** `systemctl disable yashigani.service` + unit file deleted + `daemon-reload`
- **Linux rootless Podman:** `systemctl --user disable yashigani.service` + unit file deleted + `daemon-reload`
- **loginctl disable-linger:** runs only when `--remove-volumes` is passed, as
  part of the full-clean exit path that removes data and linger together. A
  plain `uninstall.sh` preserves linger so a re-install picks up the user
  systemd instance cleanly without requiring manual re-enablement.
- **macOS:** `launchctl unload` + plist deleted
- **Docker Linux:** nothing to remove (no unit was written)

### Security model: SAML SP key must be RSA (ACS-RISK-044)

SAML SP key generation is now explicit and validated. `install.sh` generates
`docker/secrets/saml_sp.key` (RSA-4096) + `docker/secrets/saml_sp.crt` at
install time and immediately asserts the generated key is RSA — failing the
install if the invariant is not met.

Runtime enforcement was added in parallel (Tom): `SAMLProvider.__init__` calls
`_assert_rsa_sp_key()` in `src/yashigani/sso/saml.py`, which uses
`cryptography.hazmat` to confirm `isinstance(key, RSAPrivateKey)`. Any non-RSA
SP key disables SAML at startup, fail-closed.

**Rationale:** ACS-RISK-044 / CVE-2026-41989 — libgcrypt ECDH heap-buffer-overflow.
RSA SP keys never enter the ECDH `gcry_pk_decrypt` path in libgcrypt. EC keys
do. The RSA requirement makes the CVE dead code on a standard Yashigani install.

BYOK admins must generate SP keys with `openssl genrsa`. See `docs/yashigani_install_config.md` §8.2.

**PQR forward note:** When ML-KEM/Kyber key-transport is standardised in the
SAML 2.0 / xmlsec / IdP ecosystem, the RSA requirement will be revisited per the
ACS-RISK-044 forward-tracking note in the risk register.

## Known limitations

### Iptables FORWARD policy for rootful Podman installs

On Linux hosts running rootful Podman, traffic to the install's external HTTP/HTTPS ports is delivered via iptables DNAT to the container. If the host's iptables `FORWARD` chain has policy `DROP` and no matching `ACCEPT` rule for the DNATed traffic, external clients will see TCP timeouts to the install's ports. The install will converge green inside the VM — this is a network-reachability precondition, not an installer bug.

Check: `sudo iptables -L FORWARD -n | head -5` — if `policy DROP` and dropped-packet counters increase when you `curl` from outside, this is your symptom.

Remediation: add an explicit `ACCEPT` rule for the container subnet, or set `sudo iptables -P FORWARD ACCEPT`. Alternatively, use `--http-port 8080 --https-port 8443` (new in this release) to bind on higher ports if the operator's firewall configuration restricts 80/443.

Full details and the minimum-permissive iptables rule: `docs/yashigani_install_config.md §1.3`.

### macOS Podman rootless — customer-flow not empirically validated

The virtiofs `:U` secrets-mount fix (PRs #135, #143) is structurally complete.
`docker-compose.podman-virtiofs-override.yml` applies `:U` to all 16 secrets
bind-mounts on the macOS Podman path; Linux Podman is unaffected.

The admin bootstrap flow (admin1 + admin2 login) has been proven reproducible on
macOS Podman rootless (Track B v4 screenshots: 2026-05-13). The full
alice→agent→LLM customer flow has **not** been empirically exercised on this
runtime in v2.23.4. This is the same status as all other runtimes — the
customer-flow gap is tracked separately from the macOS Podman PKI fix.

**Developer workstation deployment caveat:** macOS Podman rootless is a
supported but non-production runtime. The production target remains Linux
rootful Podman (VM) and Kubernetes (Helm). macOS is suitable for local
development and demo.

TAG-CONDITIONAL removed from macOS Podman rootless. The PKI bug class is fixed.
Remaining validation gap (customer flow) is universal across runtimes, not
macOS-specific.
