## Highlights

### WebAuthn / FIDO2 Hardware-Key Admin Login

Admins can now authenticate to the Yashigani backoffice using a physical
FIDO2 hardware key (YubiKey 5 series, Security Key NFC, Titan Key, any FIDO2
authenticator) in addition to, or instead of, entering a TOTP code.

**OWASP ASVS V2.8 compliance.** Phishing-resistant, hardware-bound credentials
with sign_count replay protection. Cross-origin attestation is rejected at the
server level.

### Licence expiry UX + grace period (Phase 2)

Operators now have explicit, progressive notification of licence expiry across every operator-facing surface. The backoffice banner, a new machine-readable API endpoint, and a deterministic grace-period enforcement model replace the previous silent-fallback-to-Community behaviour.

#### Banner system

Every backoffice admin page shows a licence expiry banner when the expiry is within 30 days. Banner severity escalates automatically:

| Days relative to expiry | Mode | Severity |
|---|---|---|
| >30 days | `active` | Hidden |
| 7–30 days | `warning` | Yellow — "Yashigani licence expires in X days — renew to avoid service interruption." |
| 1–7 days | `critical` | Orange — "...renew now, sales@agnosticsec.com." |
| 0 days (expires today) | `critical` | Orange |
| 0–14 days past expiry | `expired` | Red — grace period remaining shown |
| 14–30 days past | `readonly` | Red — "Gateway is in read-only mode" |
| 30+ days past | `blocked` | Red — "Gateway is blocked" |

#### Grace-period enforcement

Post-expiry, Yashigani degrades service progressively instead of cutting off immediately:

- **Days 0–14 past expiry (EXPIRED / grace):** full service continues. Red banner on all pages. Daily audit log entry written.
- **Day 14–30 (READONLY):** operators can view all admin pages but cannot change configuration; new agent-runs are blocked. In-flight agent runs complete. HTTP 403 returned for write operations.
- **Day 30+ (BLOCKED):** HTTP 503 returned on all data-plane requests with `Retry-After` header and renewal URL.

This is a fail-explicit, not fail-silent, design: each mode is logged, audited, and surfaced to the operator, consistent with `feedback_zero_trust_default.md`.

#### Machine-readable expiry status API

New endpoint available to monitoring scripts and CLI tools:

```
GET /api/v1/license/status          (also at GET /admin/license/status)
Authorization: admin session cookie

Response:
{
  "valid": true,
  "expires_at": "2026-12-01T00:00:00+00:00",
  "days_remaining": 208,
  "grace_period_active": false,
  "mode": "active"
}
```

`mode` values: `"active"` | `"warning"` | `"critical"` | `"expired"` | `"readonly"` | `"blocked"`.

Authentication required (admin session). Accessible in READONLY mode (read-only-safe path).

#### Daily grace-period audit events

When in `expired`, `readonly`, or `blocked` mode, the scheduler emits one audit event per calendar day with:
- `event_type: "LICENSE_GRACE_PERIOD"`
- `component` field: mode, days-since-expiry, expires_at

Provides an auditable trail for GRC / compliance purposes without flooding logs.

## Changes

### New Features

#### F-V233-001 — WebAuthn / FIDO2 hardware-key admin login

- **New Alembic migration `0007_webauthn_admin_creds`:** extends the existing
  `webauthn_credentials` table with `admin_id` (FK to `admin_accounts`),
  `transports` (CTAP transport hints array), `friendly_name` (operator label),
  and upgrades `sign_count` from `INTEGER` to `BIGINT`.

- **Postgres + Redis backed WebAuthn service (`pg_webauthn.py`):**
  replaces the v0.9.0 in-memory credential store with durable Postgres
  storage. Challenges use Redis with 5-minute TTL and atomic GETDEL
  (single-use guarantee). Public keys are encrypted at rest via
  `pgp_sym_encrypt`.

- **New API endpoints (`/api/v1/admin/webauthn/`):**

  | Method | Path | Auth | Notes |
  |---|---|---|---|
  | POST | `/api/v1/admin/webauthn/register/start` | Session | Begin registration |
  | POST | `/api/v1/admin/webauthn/register/finish` | Session | Complete registration |
  | POST | `/api/v1/admin/webauthn/login/start` | **Public** | Begin authentication |
  | POST | `/api/v1/admin/webauthn/login/finish` | **Public** | Complete + issue session |
  | GET | `/api/v1/admin/webauthn/credentials` | Session | List credentials |
  | DELETE | `/api/v1/admin/webauthn/credentials/{id}` | Session + **step-up** | Revoke |

- **Login page update:** the admin login page (`/admin/login`) shows a
  "Sign in with Security Key" button when `navigator.credentials` is
  available. Loaded via external JS (`/static/js/webauthn_login.js`),
  CSP-compliant (no inline scripts, no unsafe-inline).

- **Recovery escape hatch:** password + TOTP login is never disabled while
  WebAuthn is configured. Lost all keys? Use the TOTP form as normal.

- **Step-up required for credential revocation (ASVS V6.8.4):** `DELETE
  /api/v1/admin/webauthn/credentials/{id}` requires a fresh TOTP step-up
  (within `YASHIGANI_STEPUP_TTL_SECONDS`, default 5 minutes). Prevents a
  stolen session cookie from silently removing all keys.

- **Audit events:**
  - `WEBAUTHN_CREDENTIAL_REGISTERED` — new key registered
  - `WEBAUTHN_LOGIN_SUCCESS` — successful hardware-key login
  - `WEBAUTHN_LOGIN_FAILURE` — failed assertion (wrong key, sign_count rollback, etc.)
  - `WEBAUTHN_CREDENTIAL_REVOKED` — credential revoked by admin

### Security

- sign_count rollback detection: new_sign_count ≤ stored (when stored > 0) is
  rejected as a cloned-authenticator indicator (ASVS V2.8, CWE-287).
- Enumerate-safe responses: unknown username on `login/start` returns the same
  `no_credentials_registered` 400 as a known user with no keys.
- Cross-origin attestation rejected by `py_webauthn` via `expected_origin`
  parameter derived from `X-Forwarded-Proto` + `Host` headers.

### Configuration

New environment variables (all optional — sensible defaults for local install):

```dotenv
YASHIGANI_WEBAUTHN_RP_ID=admin.example.com
YASHIGANI_WEBAUTHN_RP_NAME=Yashigani Backoffice
YASHIGANI_WEBAUTHN_USER_VERIFICATION=preferred
YASHIGANI_WEBAUTHN_ATTESTATION=none
YASHIGANI_WEBAUTHN_REQUIRE_RESIDENT_KEY=false
```

See `docs/yashigani_install_config.md` § 19 for full configuration reference.

## Upgrade Notes

1. Run `alembic upgrade head` (or restart the stack — the lifespan runs
   `run_migrations()` automatically).
2. Migration `0007` alters the `webauthn_credentials` table — no data loss,
   all existing rows are preserved.
3. Set `YASHIGANI_WEBAUTHN_RP_ID` to your domain before the first registration.
4. TLS is required for WebAuthn — ensure Caddy is configured correctly.
5. **Remove `YASHIGANI_WEBAUTHN_ORIGIN` from your `.env` file** — this variable
   is no longer read. The WebAuthn origin is now derived automatically from the
   `X-Forwarded-Proto` and `Host` headers on each request (set by Caddy). If you
   leave the variable set it will be silently ignored; removing it avoids operator
   confusion on future upgrades. Previous variables `YASHIGANI_WEBAUTHN_RP_ID` and
   `YASHIGANI_WEBAUTHN_RP_NAME` remain required and unchanged.
